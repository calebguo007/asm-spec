# Agent Service Manifest — 调研文档

> **One-liner**: OpenAPI describes what a service *can do*. ASM describes what a service *is worth*. — **"OpenAPI for agents, but for value, not capability."**

> 核心问题：Agent 在面对多个可选服务时，缺少一套标准化的"服务标签体系"来做机器可读的评估和自动化采购决策。
> 调研日期：2026-03-20

---

## 一、问题定义

### 场景

Agent 接到任务（如"帮我把这段视频加字幕"），面前有多个可选服务：
- 服务 A：$0.10/分钟，质量 90 分，延迟 2 秒
- 服务 B：$0.03/分钟，质量 75 分，延迟 10 秒
- 服务 C：免费，要排队 5 分钟，质量 60 分

### 当前的问题

Agent 没法做结构化比较。因为：
1. **定价方式不统一**：按 token / 按次 / 按分钟 / 按字数 / 按结果
2. **能力描述不统一**：每个服务自说自话，没有可比性
3. **质量没有标准度量**：好不好全靠用户口碑，agent 无法自动评估
4. **SLA 信息不暴露**：延迟、可靠性、并发限制等关键信息不是机器可读的

### 我想做的

一套**度量衡标准**（暂称 Agent Service Manifest，ASM）——让所有服务按统一维度暴露自己的"价值"。这个"价值"是多维向量（成本、质量、能力范围、可靠性……），Agent 拿到结构化信息后，根据用户偏好函数自动选择，然后完成购买。

### 核心创新

不是支付本身，是**支付之前的决策协议**——让"服务的价值"变得机器可读、可比较、可自动决策。

### 没有 ASM 会怎样？— 失败案例

> 以下是一个**真实会发生**的场景，用来证明"这个空白正在造成伤害"。

**场景**：用户让 agent 做一段 10 分钟的视频字幕。

**没有 ASM 的世界**（现状）：

```
Agent 收到任务 → 搜索"subtitle API" → 找到 3 个服务
→ 看不到质量评分，看不到 SLA，定价单位都不一样
→ 只能按名字选（选了最有名的 A）或按价格选（选了最便宜的 C）

结果 1（按名气选 A）：
  花了 $1.00，质量 90 分，2 秒出结果 ✅ 但贵了 3 倍
  — 其实服务 B 的 $0.30 + 质量 75 分对这个任务完全够用

结果 2（按价格选 C）：
  免费，但排队 5 分钟，质量 60 分，错别字连篇 ❌
  — 用户不满意，agent 又调了一次 A 来修，总成本 $1.00 + 5 分钟
```

**有 ASM 的世界**：

```
Agent 收到任务 → 查询 ASM Registry → 拿到 3 个服务的结构化清单
→ 用户偏好函数：quality ≥ 0.7 AND cost ≤ $0.50 AND latency ≤ 30s
→ 自动匹配 → 选 B

结果：$0.30，质量 75 分，10 秒出结果 ✅
  — 省了 70% 成本，质量达标，无需返工
```

**对比**：

| | 没有 ASM | 有 ASM |
|---|---|---|
| 决策方式 | 按名气猜 / 按价格赌 | 结构化匹配 |
| 成本 | $1.00（选贵了）或 $1.00+5min（选错返工） | $0.30 |
| 质量 | 过剩（90分）或不足（60分） | 刚好达标（75分） |
| 时间 | 2s 或 300s+ | 10s |
| **agent 能力** | **不能自主决策，只能碰运气** | **可以自主决策，可解释** |

> 💡 **关键洞察**：没有 ASM，agent 的"智能"在服务选择这一步归零——不管你用多强的模型，面对非结构化的服务信息，它和随机选没有本质区别。

## 二、现有协议层的空白分析

### MCP（Model Context Protocol）— Anthropic

**版本**：2025-03-26

**Tool Definition Schema**：
```json
{
  "name": "get_weather",
  "description": "Get current weather for a location",
  "inputSchema": { "type": "object", "properties": {...} },
  "annotations": { /* 可选元数据 */ }
}
```

**annotations 字段（ToolAnnotations）的全部定义**：
| 字段 | 类型 | 默认值 | 含义 |
|------|------|--------|------|
| `readOnlyHint` | boolean | false | 是否只读 |
| `destructiveHint` | boolean | true | 是否有破坏性 |
| `idempotentHint` | boolean | false | 是否幂等 |
| `openWorldHint` | boolean | true | 是否与外部实体交互 |
| `title` | string | — | 人类可读标题 |

**关键发现**：
- ❌ **没有 pricing/cost 字段**
- ❌ **没有 quality/accuracy 字段**
- ❌ **没有 SLA/latency/uptime 字段**
- ❌ **没有 capability scoring 字段**
- ❌ GitHub 仓库（modelcontextprotocol/specification）中**零个** issue/discussion 涉及 pricing、cost、marketplace、billing、service discovery
- annotations 是 hint 性质的，协议明确说"不能基于不受信任服务器的 annotations 做决策"

**结论：MCP 定义了"这个工具能做什么"，完全没覆盖"这个工具值多少"。**

### AP2（Agent Payments Protocol）— Google

**版本**：V0.1（2025）

**设计哲学（值得复用的部分）**：
1. **角色解耦**：用户 / 购物代理 / 凭证提供商 / 商家端点 / 支付处理器 — 职责分明
2. **可验证数字凭证（VDC）**：加密签名的授权链，不依赖概率性 LLM 输出
3. **从"命令"到"契约对话"**：消息不是 API 指令，是可审计的协商步骤
4. **Agent Card**：JSON 格式声明代理的 capabilities、skills、url
5. **Intent Mandate**：用户不在场时的预授权（含自然语言描述 + 有效期 + 签名）

**AP2 解决的问题**：怎么安全地付钱
**AP2 不解决的问题**：怎么选服务、服务值不值得买

**结论：AP2 是交易协议，不是评估协议。但它的架构设计（角色解耦、VDC、Agent Card）值得借鉴。**

### A2A（Agent-to-Agent）— Google

解决 agent 间通信。不涉及服务定价标准化。

### 空白地图

```
MCP  → "这个工具能做什么"     ✅ 已解决
A2A  → "agent 之间怎么通信"    ✅ 已解决
AP2  → "怎么安全地付钱"        ✅ 已解决（V0.1）

ASM  → "这个服务值多少、和别的比怎么样、该不该买"  ❌ 无人做
```

**ASM 是 MCP 和 AP2 之间缺失的一层。**

> OpenAPI tells agents what a service *can do*. ASM tells agents what a service *is worth*.
> — OpenAPI for agents, but for value, not capability.

## 三、现有最接近的产品分析

### OpenRouter

**做了什么**：
- 多 LLM API 聚合（OpenAI / Anthropic / Google / Meta / Mistral 等）
- 统一 API 格式（兼容 OpenAI Chat API）
- 每次响应返回 `usage.cost` 字段（实际花费）
- 自动路由：选成本最低的 GPU，5xx 自动回退

**没做什么**：
- ❌ 只覆盖 LLM 推理这一个品类
- ❌ 没有质量评分（不告诉你哪个模型质量更好）
- ❌ 路由决策不透明（用户无法设定偏好函数）
- ❌ 不是协议标准，是封闭平台

**瓶颈**：OpenRouter 是"LLM 版大众点评"，但它只管一个品类，而且评估维度不够结构化。

### 硅基流动

**做了什么**：
- 国内多 LLM API 聚合
- 统一调用接口 + 统一计费
- 价格列表在网页上给人看

**没做什么**：
- ❌ 同 OpenRouter，只覆盖 LLM 推理
- ❌ 没有机器可读的评估数据
- ❌ 没有 agent 自动选择的 API

### GPT Store / OpenAI

**做了什么**：
- GPTs 应用市场（用户可浏览、使用）
- 有排行榜和分类

**没做什么**：
- ❌ 这是给人浏览的 UI，不是给 agent 查询的 API
- ❌ 没有标准化的能力/定价/质量描述
- ❌ 不支持 agent 自动评估和采购

### Skyfire

**做了什么**：
- "Agentic Commerce Platform"
- KYA（Know Your Agent）身份验证
- Agent 自主支付（USDC 稳定币）

**没做什么**：
- ❌ 侧重支付执行，不是服务评估
- ❌ 走 crypto 路线，主流采用率存疑

## 四、巨头生态布局分析

### 最应该嵌入谁的生态？

| 巨头 | 已有协议 | 空白 | 适合嵌入点 |
|------|---------|------|-----------|
| **Anthropic** | MCP（工具连接） | MCP tool schema 没有 pricing/quality | **在 MCP annotations 里扩展 pricing/quality 字段** |
| **Google** | A2A + AP2（通信+支付） | AP2 有 Agent Card 但不含服务评估 | **在 Agent Card 里扩展 service manifest** |
| **OpenAI** | GPT Store + Agent SDK + 已支持 MCP | 没有 agent 自动评估服务的机制 | 通过 MCP 扩展间接覆盖 |
| **Stripe** | Agent Toolkit（支付集成） | 只做支付，不做评估 | 不合适 |

**判断：Anthropic（MCP 扩展）和 Google（AP2/A2A 扩展）是最合适的两个嵌入点。**

理由：
1. MCP 已经是事实标准（OpenAI、Google 都支持），在它的 tool schema 里加 pricing/quality 字段影响面最大
2. AP2 的 Agent Card 设计最接近"服务清单"的概念，天然适合扩展
3. 两条路不冲突，可以同时推

## 五、ASM 协议设计（v0.2）

> 从"极早期草案"升级为"可跨品类验证的 v0.2"。用 5 大品类（LLM / 图像生成 / 视频生成 / 语音合成 / GPU 计算）的真实定价结构倒推 schema 设计。

### 5.1 设计约束（对齐 MCP 设计原则）

ASM 必须兼容 MCP 的 8 条设计原则，否则 SEP 提案不会被接受：

| MCP 原则 | ASM 的应对 |
|---------|----------|
| **组合而非特定** | ASM 是一个通用的 annotation 扩展，不为特定品类设计字段 |
| **互操作性而非优化** | 所有字段可选，能力协商：服务只暴露它能暴露的维度 |
| **稳定性而非速度** | 只定义核心原语（pricing/quality/sla），品类特定维度通过 `extensions` 命名空间扩展 |
| **实证而非辩论** | 下方给出 3 个跨品类的真实 demo 证明可行性 |
| **标准化而非创新** | pricing 结构参照 OpenRouter/Replicate/Vertex AI 已有惯例，不发明新范式 |
| **实用而非纯粹** | 允许 `estimated: true` 标记非精确数据，承认现实不完美 |

### 5.2 分类体系 (Taxonomy)

为了让 agent 能跨品类搜索和匹配，ASM 定义一套标准化的服务分类：

```
taxonomy: "<domain>.<category>[.<subcategory>]"
```

**初始分类树**（基于当前 AI 服务生态）：

```
ai.llm.chat                    # 对话式 LLM（GPT-4o, Claude, Gemini）
ai.llm.completion               # 补全式 LLM
ai.llm.embedding                # 向量嵌入（text-embedding-3-large）
ai.vision.image_generation      # 文生图（FLUX, DALL-E, Imagen）
ai.vision.image_editing         # 图像编辑（inpainting, upscale）
ai.vision.ocr                   # 文字识别
ai.video.generation             # 文/图生视频（Veo, Wan, Kling）
ai.video.subtitle               # 字幕生成
ai.video.editing                # 视频编辑
ai.audio.tts                    # 文字转语音（ElevenLabs, OpenAI TTS）
ai.audio.stt                    # 语音转文字（Whisper, Deepgram）
ai.audio.music                  # 音乐生成（Lyria, Suno）
ai.code.generation              # 代码生成
ai.code.review                  # 代码审查
ai.data.extraction              # 数据提取/结构化
ai.data.search                  # 搜索/检索（Grounding, RAG）
infra.compute.gpu               # GPU 计算资源（Replicate, RunPod）
infra.storage.object             # 对象存储
infra.storage.vector             # 向量数据库
```

**设计决策**：
- 三级结构（domain.category.subcategory），足够精确但不过度碎片化
- agent 可以用前缀匹配：查 `ai.vision.*` 即可拿到所有视觉相关服务
- 分类树本身可通过 ASM Registry 动态扩展，不硬编码在协议中

### 5.4 设计原则（v0.2 更新）

1. **MCP 兼容**：ASM 可以作为 MCP tool annotations 的 `x-asm` 扩展字段嵌入
2. **多维不强制**：只有 `asm_version`、`service_id`、`taxonomy` 是必填，其余全部可选
3. **多计费维度并存**：`billing_dimensions` 是数组，一个服务可以同时有 input_token + output_token + image 三种计费
4. **条件定价**：支持 `conditions`（如"context >200K 价格翻倍"）和 `tiers`（阶梯价）
5. **质量引用第三方基准**：`self_reported` 字段区分自评 vs 第三方验证
6. **支付声明不执行**：ASM 暴露 `ap2_endpoint`，agent 拿到后走 AP2 协议完成交易
7. **可机器比较**：agent 拿到多个 ASM 后，用偏好函数自动排序
8. **扩展不污染核心**：品类特定字段放 `extensions` 命名空间，核心 schema 永远稳定

### 最小决策函数 Demo — "这东西真的可以算"

> 以下用最简单的数学证明：有了 ASM，agent 的服务选择可以变成一个**可计算的优化问题**。

#### 偏好函数

```python
def score(service, weights):
    """
    用户设定权重，agent 自动打分。
    所有维度归一化到 [0, 1]，越高越好。
    """
    return (
        weights["cost"]    * (1 - normalize(service.cost, min=0, max=max_cost))  # 越便宜越好
      + weights["quality"] * service.quality_score                                # 越高越好
      + weights["speed"]   * (1 - normalize(service.latency, min=0, max=max_lat)) # 越快越好
    )
```

#### 实际计算

三个字幕服务的 ASM 数据：

| 服务 | cost ($/min) | quality | latency (s/min) |
|------|-------------|---------|-----------------|
| A    | 0.10        | 0.92    | 2               |
| B    | 0.03        | 0.75    | 10              |
| C    | 0.00        | 0.60    | 300             |

归一化（min-max 到 [0,1]，cost 和 latency 反转）：

| 服务 | norm_cost | quality | norm_speed |
|------|-----------|---------|------------|
| A    | 0.00      | 0.92    | 1.00       |
| B    | 0.70      | 0.75    | 0.97       |
| C    | 1.00      | 0.60    | 0.00       |

**场景 1：用户偏好"省钱优先"**（w_cost=0.5, w_quality=0.3, w_speed=0.2）

```
score(A) = 0.5×0.00 + 0.3×0.92 + 0.2×1.00 = 0.476
score(B) = 0.5×0.70 + 0.3×0.75 + 0.2×0.97 = 0.769  ← 最优
score(C) = 0.5×1.00 + 0.3×0.60 + 0.2×0.00 = 0.680
```

→ **选 B**。便宜 + 质量够用 + 速度快。

**场景 2：用户偏好"质量第一"**（w_cost=0.1, w_quality=0.7, w_speed=0.2）

```
score(A) = 0.1×0.00 + 0.7×0.92 + 0.2×1.00 = 0.844  ← 最优
score(B) = 0.1×0.70 + 0.7×0.75 + 0.2×0.97 = 0.689
score(C) = 0.1×1.00 + 0.7×0.60 + 0.2×0.00 = 0.520
```

→ **选 A**。质量最高，虽贵但用户愿意付。

**场景 3：用户偏好"免费就行"**（w_cost=0.9, w_quality=0.05, w_speed=0.05）

```
score(A) = 0.9×0.00 + 0.05×0.92 + 0.05×1.00 = 0.096
score(B) = 0.9×0.70 + 0.05×0.75 + 0.05×0.97 = 0.716
score(C) = 0.9×1.00 + 0.05×0.60 + 0.05×0.00 = 0.930  ← 最优
```

→ **选 C**。免费，质量差但用户不在意。

#### 关键证明

1. **同样的 3 个服务，不同用户偏好 → 不同最优解** — 这不是"总有一个最好的"，而是"最好的取决于你是谁"
2. **决策完全可解释** — agent 可以告诉用户"我选了 B，因为你的偏好是省钱优先，B 的综合得分 0.769 最高"
3. **决策可自动化** — 不需要人参与，agent 查 ASM → 算分 → 选服务 → 调 AP2 付款，全链路自动
4. **决策可审计** — 权重、得分、选择理由全部可追溯，出问题可以回查

> 🎯 **这就是 ASM 的核心价值**：把"哪个服务更好"从一个主观判断变成一个可计算、可解释、可自动化的数学问题。

## 九、Problem Statement（SEP-ready 版本）

> 这一节按 MCP SEP 的 8 个必需章节格式预写，Day 5 写 RFC 时可直接复用。

### 9.1 Abstract（~200 字）

The Model Context Protocol enables agents to discover and invoke external tools, but provides no standard mechanism for agents to evaluate, compare, or select among competing services that offer similar capabilities. When multiple services can fulfill a task (e.g., three different subtitle APIs), agents lack structured information about pricing, quality, latency, and reliability — forcing them into blind selection that is equivalent to random choice regardless of model intelligence.

We propose **Agent Service Manifest (ASM)**, an extension to MCP's `ToolAnnotations` that allows service providers to declare standardized, machine-readable value descriptors: pricing dimensions, quality benchmarks, SLA guarantees, and payment methods. ASM is not a payment protocol — it is the **pre-payment decision layer** that makes service value computable, comparable, and automatically actionable by agents.

One-liner: *"OpenAPI describes what a service can do. ASM describes what a service is worth."*

### 9.2 Motivation（为什么现有协议不够）

**现状**：

1. **MCP ToolAnnotations 只描述行为**：`readOnlyHint`、`destructiveHint`、`idempotentHint` 等 5 个字段全部是行为提示，没有任何经济或质量相关字段。协议明确声明 "annotations 不应被信任用于做决策"。

2. **定价方式碎片化**：真实 AI 服务存在至少 8 种计费模式 —— per-input-token、per-output-token、per-image、per-second-of-video、per-character、per-gpu-second、per-request、subscription+credits。Agent 无法在非结构化的自然语言描述中解析和比较这些定价。

3. **质量不可量化**：Agent 无法获得机器可读的质量数据。大部分质量信息以博客文章、推特讨论、人类口碑等非结构化形式存在。

4. **AP2 解决支付但不解决选择**：Google AP2 定义了 agent 如何安全地完成交易，但前提是 agent 已经知道要买什么。"买哪个"的决策完全空白。

**造成的伤害**（量化）：

以一个自主运行的 coding agent 为例（如 Claude Code / Cursor），它在一次复杂任务中可能需要：
- 调用 LLM（3+ 个可选模型）
- 生成图像（5+ 个可选服务）
- 运行代码（3+ 个 GPU 平台）

如果每一步都盲选最贵的服务，成本可能比最优选择高 **3-10 倍**。如果盲选最便宜的，质量不达标导致返工，总成本反而更高。

**这不是效率优化，是范式缺失** —— agent 经济体正在形成，但连最基本的"商品标签"都没有。

### 9.3 Specification（技术规范）

见上方 [第五章：ASM 协议设计 v0.2](#五asm-协议设计v02)。

核心要点：
- JSON Schema 格式，兼容 MCP tool schema
- 3 个必填字段（`asm_version`、`service_id`、`taxonomy`），其余全部可选
- `billing_dimensions` 数组支持多维度并存（解决 LLM 按 input+output 分别计费的问题）
- `conditions` 和 `tiers` 支持条件定价和阶梯价
- `extensions` 命名空间隔离品类特定字段
- 已用 3 个真实服务（Claude Sonnet 4 / FLUX 1.1 Pro / Veo 3.1）验证

### 9.4 Rationale（为什么这样设计）

| 设计决策 | 选了什么 | 替代方案 | 为什么选这个 |
|---------|---------|---------|------------|
| 格式 | JSON Schema | YAML / Protobuf | MCP 本身用 JSON Schema，无缝嵌入 |
| 定价模型 | `billing_dimensions` 数组 | 单一 `price_per_unit` | 真实世界的 LLM 有 input/output 两种 token 价格，必须支持多维度 |
| 质量度量 | 引用第三方 benchmark + `self_reported` 标记 | 统一打分制 | 不同品类的质量度量完全不同（LLM 用 Elo、图像用 FID、语音用 MOS），强制统一不现实 |
| 分类体系 | 三级 taxonomy | 自由标签 | 自由标签无法做前缀匹配，agent 没法搜"所有视觉类服务" |
| AP2 集成 | 声明 `ap2_endpoint` | 内嵌 AP2 支付逻辑 | ASM 是评估协议不是支付协议，职责分明 |
| 扩展机制 | `extensions` 命名空间 | 直接加核心字段 | 符合 MCP "组合而非特定" 原则，核心 schema 永远稳定 |

### 9.5 Backward Compatibility（向后兼容性）

- ASM 作为 `ToolAnnotations` 的**可选扩展**嵌入（`x-asm` 命名空间），不修改任何现有字段
- 不支持 ASM 的客户端会直接忽略 `x-asm` 字段，零破坏
- 不支持 ASM 的服务器不需要做任何改动

### 9.6 Reference Implementation（参考实现计划）

| 组件 | 描述 | 状态 |
|------|------|------|
| `asm-schema` | JSON Schema 定义文件 | ✅ v0.2 完成 |
| `asm-registry` | 一个 MCP Server，手动收录 20-30 个热门服务的 ASM | 🔜 Day 3-4 |
| `asm-scorer` | 偏好函数 + 归一化 + 排序引擎 | 🔜 Day 3-4 |
| `asm-demo` | Agent 自动选择服务的端到端 demo | 🔜 Day 3-4 |

### 9.7 Security Implications（安全影响）

1. **ASM 数据可信度**：服务方自报的 ASM 数据可能不准确。应对：
   - `self_reported` 字段区分自评 vs 第三方验证
   - `verified_by` 字段声明验证方
   - 长期方向：ASM Registry 可以做独立审计

2. **定价操纵**：服务方可能报低价吸引流量再加价。应对：
   - ASM 数据带时间戳，agent 可以检测价格变化
   - Registry 可以缓存历史价格，标记异常波动

3. **与 MCP 安全模型的兼容**：MCP 协议明确说"不能基于不受信任服务器的 annotations 做决策"。ASM 不改变这一立场——agent 使用 ASM 做的是**推荐排序**，最终决策仍需用户确认（或用户预授权的 AP2 Intent Mandate）。

## 十一、核心论文深度调研（2026-04-03 补充）

### 11.1 论文 1：A Survey of AI Agent Protocols (arXiv:2504.16736)

**核心贡献**：首个 AI Agent 协议的全面综述，提出二维分类法：
- **维度 1**：Context-Oriented（面向上下文，如 MCP）vs Inter-Agent（Agent 间，如 A2A）
- **维度 2**：General-Purpose（通用）vs Domain-Specific（特定领域）

**比较维度**：安全性、可扩展性、延迟

**识别的 Gap**：
1. Agent 与外部工具/数据源通信无统一标准
2. Agent 间协作困难
3. 扩展性受限
4. 复杂任务处理能力弱

**ASM 的定位**：
- 该综述覆盖了 MCP/A2A/ACP/ANP 等协议，但**完全没有覆盖"服务经济学"层面**
- 论文的分类体系里，ASM 属于一个**全新的第三维度**——不是"上下文"也不是"Agent 间通信"，而是"**服务价值评估**"
- ASM 论文可以在 Related Work 中引用此综述，明确说："现有协议覆盖了通信和上下文，但遗漏了经济决策层"

**可引用的数据/观点**：
- "缺乏标准化协议导致 agent 难以协同工作或有效扩展"——直接支持 ASM 的 motivation
- 二维分类法可以被扩展成三维（加入"service economics"维度），展示 ASM 的创新性

### 11.2 论文 2：Agent-as-a-Service based on Agent Network / AaaS-AN (arXiv:2505.08446)

**核心贡献**：基于 RGPS（Role-Goal-Process-Service）标准的"Agent 即服务"范式

**两个核心组件**：
1. **动态 Agent 网络**：Agent 和 Agent 组建模为顶点，根据任务和角色依赖自组织
2. **服务导向 Agent**：含服务发现、注册和互操作性协议

**编排机制**：服务调度器 + 执行图 → 分布式协调、上下文跟踪、运行时任务管理

**规模验证**：100+ Agent 服务的 MAS + 10,000 个长视界多智能体工作流数据集

**⚠️ 这是 ASM 最近的竞品，必须讲清楚区别：**

| 维度 | AaaS-AN | ASM |
|------|---------|-----|
| **核心问题** | Agent 之间怎么组织协作 | Agent 怎么选择和评估服务 |
| **标准基础** | RGPS（角色-目标-过程-服务） | MCP ToolAnnotations 扩展 |
| **关注层面** | 组织架构层（谁和谁协作、怎么编排） | 经济决策层（服务值多少、该不该买） |
| **服务发现** | 有，但侧重"发现谁能协作" | 有，侧重"发现谁性价比最高" |
| **定价信息** | ❌ 无 | ✅ 12 种计费维度 + 阶梯价 + 条件定价 |
| **质量指标** | ❌ 无标准化质量度量 | ✅ 第三方 benchmark + 自评标记 + 排行榜 |
| **SLA** | ❌ 无 | ✅ 延迟/吞吐/可用性/并发 |
| **支付集成** | ❌ 无 | ✅ 预埋 AP2 对接 |
| **偏好函数** | ❌ 无 | ✅ 用户偏好 → 归一化 → 加权排序 |
| **互补关系** | AaaS-AN 编排多个 Agent 协作完成任务 | ASM 帮每个 Agent 在执行中选择最优服务 |

**一句话区分**：AaaS-AN 解决的是"Agent 团队怎么协作"，ASM 解决的是"每个 Agent 怎么花钱"。两者是不同层面的问题，可以互补——Agent 通过 AaaS-AN 分配任务后，每个子任务用 ASM 选择最优服务执行。

**Discord 回复可用的话术**：
> "AaaS-AN is about organizational collaboration (which agents work together). ASM is about economic evaluation (which service to buy). They're complementary layers — AaaS-AN orchestrates the team, ASM optimizes each team member's purchasing decisions."

### 12.4 Agent Receipts SDK — Signed Receipts 技术已成熟

**来源**：[agent-receipts/sdk-py](https://github.com/agent-receipts/sdk-py/)（已迁移至 [agent-receipts/ar](https://github.com/agent-receipts/ar)）

**核心能力**：
- 创建、签名、哈希链、存储和验证 AI Agent 操作的加密签名记录
- 遵循 W3C 可验证凭证标准，使用 Ed25519 签名
- 支持哈希链（hash chain），确保日志不可篡改

**Agent Receipts 数据模型**：
- `Issuer`：执行操作的 Agent ID
- `Principal`：授权者（人或组织）
- `Action`：发生了什么（类型 + 风险级别）
- `Outcome`：成功/失败/可逆性
- `Chain`：序列号 + 前一个 receipt 的哈希 → 形成不可篡改链

**与 ASM 的互操作方案（具体技术设计）**：

```
完整信任链路：

1. Provider 声明 ASM manifest
   → pricing, quality, SLA, payment

2. Agent 查询 ASM Registry → 用偏好函数选择服务
   → "我选了 FLUX 1.1 Pro，因为 cost=0.04/image, quality=4.8 FID"

3. Agent 调用服务 → 服务执行并返回结果

4. Agent Receipts SDK 生成 signed receipt
   → action: "ai.vision.image_generation"
   → 记录实际花费、实际延迟、实际质量

5. 对比 ASM 声明 vs Receipt 实际
   → trust_score = f(declared_quality, actual_quality, declared_latency, actual_latency)
   → Provider 信誉分数动态更新
```

**ASM schema 中需要预埋的字段**：
- `receipt_endpoint`：Provider 的 receipt 验证端点
- `receipt_format`：支持的 receipt 格式（如 `w3c_verifiable_credential`）
- `verification.method`：验证方式（如 `signed_receipt_comparison`）

**合作方向**：
- 联合写 paper：ASM + Receipts = 完整的 agent 服务信任体系
- 联合做 demo：ASM 选服务 → 执行 → Receipt 验证 → 信誉更新
- 联合提 SEP：在 MCP 社区一起推"service economics + trust verification"

## 十三、六大核心问题深度思考（2026-04-03）

> 这些是 ASM 从"好 idea"变成"能落地的协议"必须回答的问题。

### 13.1 Provider 为什么愿意填 ASM？— 采纳激励分析

**问题**：如果没人填 ASM，协议就是空架子。Provider 为什么要主动暴露自己的定价和质量？

**答案：不是靠"道德号召"，而是靠"利益驱动"。**

参考 API Marketplace 的历史经验（RapidHub、AWS Marketplace、OpenRouter），Provider 填写结构化信息的动机有 3 层：

| 层级 | 激励 | 类比 |
|------|------|------|
| **L1：被发现** | 不填 ASM = Agent 找不到你 = 流量归零 | 就像不上架 App Store = 没人下载 |
| **L2：赢竞价** | Agent 用偏好函数选服务，填了 ASM 的服务才能参与排序。不填 = 永远不会被推荐 | 就像 Google Ads 不出价 = 不展示 |
| **L3：定价权** | 填了 ASM 的 Provider 可以主动定位自己（"我是最便宜的"/"我是最高质量的"），不填的只能被动等待 | 就像 Amazon 商品没价格标签 = 用户直接跳过 |

**冷启动策略**（不需要所有 provider 都填）：
1. **Phase 1：Registry 主动填充** — ASM Registry 自己去公开渠道抓取定价信息（API 文档、定价页面），Provider 不需要做任何事。这就是你 Day 3-4 要做的"手动收录 20-30 个服务"
2. **Phase 2：Provider 自愿认领** — 当 Registry 有了流量，Provider 会主动来认领和更新自己的 manifest（"你们的数据不对，我来纠正"）
3. **Phase 3：Provider 主动声明** — 当 ASM 成为事实标准，新 Provider 上线时会主动发布 `.asm.json`

**论文/提案中的表述**：
> "ASM does not require provider opt-in to bootstrap. The registry can be populated by crawling public pricing pages, similar to how search engines index the web without website owner cooperation. Provider-declared manifests are preferred but not required."

**关键设计决策**：ASM manifest 的 `self_reported: false` 标记意味着"这不是 provider 自己说的，是我们从公开数据抓取的"。这降低了冷启动门槛。

### 13.2 质量数据造假怎么办？— 信任机制设计

**问题**：Provider 自报质量 0.99、延迟 100ms，实际质量 0.6、延迟 2s，怎么办？

**答案：三层防线。**

| 层级 | 机制 | 效果 |
|------|------|------|
| **L1：标记透明度** | `self_reported: true/false` 字段 | Agent 知道"这是自评还是第三方"，可以给自评数据打折 |
| **L2：第三方基准引用** | `benchmark` + `benchmark_url` + `evaluated_at` | 引用公开排行榜（LMSYS Arena、HuggingFace Open LLM），可独立验证 |
| **L3：Signed Receipts 事后验证** | ASM 声明 vs Receipt 实际 → trust_score | 长期积累后，造假 Provider 的信誉分会持续下降 |

**L3 是最强防线**，也是和 Agent Receipts 合作的核心价值：

```
trust_score(provider) = f(
  Σ(declared_quality - actual_quality) over N transactions,
  Σ(declared_latency - actual_latency) over N transactions,
  Σ(declared_price - actual_price) over N transactions
)
```

如果 Provider 声称质量 0.95 但实际交付 0.7，trust_score 会持续下降，Agent 的偏好函数会自动降权这个 Provider。

**与加密可验证性的关系**：
参考论文 [A Framework for Cryptographic Verifiability of End-to-End AI Pipelines](https://arxiv.org/abs/2503.22573)（ACM IWSPA 2025），该论文提出用密码学证明绑定 AI 输出，使其可独立验证。ASM + Receipts 的信任链路本质上是同一思路的协议层实现：
- 该论文解决"AI 输出是否真的来自声称的模型"
- ASM + Receipts 解决"AI 服务是否真的达到了声称的质量"

**论文中可引用**：
> "ASM addresses the trust problem through a three-layer mechanism: transparency marking (self_reported flag), third-party benchmark referencing, and post-hoc verification through signed execution receipts. This approach mirrors the cryptographic verifiability framework proposed by Balan et al. (2025) for end-to-end AI pipelines, applied at the service economics layer."

### 13.3 动态定价怎么处理？— 时效性设计

**问题**：GPU 价格按需波动、LLM 价格经常调整、促销活动随时开始。ASM 的静态 JSON manifest 怎么应对？

**答案：ASM 不需要实时，只需要"足够新"。**

**核心洞察**：Agent 做的是"选择决策"，不是"精确计费"。选择阶段只需要知道"大概多少钱"就够了，精确账单在交易时由支付协议（AP2/Stripe）处理。

**设计方案**：

| 机制 | 实现 | 适用场景 |
|------|------|---------|
| **`updated_at` 时间戳** | manifest 顶层字段，标记最后更新时间 | Agent 判断数据是否过期（>24h 可降权）|
| **`estimated: true`** | 已有字段 | 标记"此价格为估算值"（如 GPU 按需定价）|
| **`pricing_url`** | 新增字段，指向实时定价 API | Agent 需要精确价格时调用 |
| **`ttl`** | 新增字段，manifest 的建议缓存时间（秒）| Registry 知道多久该刷新 |
| **Webhook 通知** | Provider 价格变化时通知 Registry | Registry 自动更新，无需轮询 |

**分层策略**：
```
快决策（95% 场景）：直接用 manifest 里的静态价格 → 足够选出最优服务
精决策（5% 场景）：Agent 调用 pricing_url 获取实时价格 → 大额交易前确认
```

**需要在 schema v0.3 中新增的字段**：
```json
{
  "updated_at": "2026-04-03T14:00:00Z",
  "ttl": 86400,
  "pricing": {
    "pricing_url": "https://api.anthropic.com/v1/pricing",
    ...existing fields...
  }
}
```

### 13.4 和 MCP 的技术集成深度 — `x-asm` vs 独立端点

**问题**：ASM 应该作为 MCP ToolAnnotations 的 `x-asm` 扩展嵌入，还是作为独立的 `.well-known/asm` 端点？

**答案：两者都做，但推进顺序不同。**

**MCP ToolAnnotations 的现状**：
- 只有 5 个标准字段（title, readOnlyHint, destructiveHint, idempotentHint, openWorldHint）
- 注解明确定义为"hints"——"不应基于不受信任服务器的 annotations 做决策"
- **MCP 规范目前没有定义 `x-` 前缀的扩展命名空间机制**

这意味着：
1. `x-asm` 在当前 MCP 规范里**没有官方支持**——客户端不会自动解析未知的 annotation 字段
2. 但 MCP 的 JSON Schema 是 `additionalProperties: true`，所以加 `x-asm` 不会破坏解析

**推荐策略**：

| 阶段 | 集成方式 | 优势 | 劣势 |
|------|---------|------|------|
| **Phase 1（现在）** | 独立的 `.well-known/asm` 端点 | 不依赖 MCP 规范变化，立即可用 | 需要 Agent 额外发一次请求 |
| **Phase 2（SEP 通过后）** | `x-asm` 嵌入 ToolAnnotations | 零额外请求，和工具定义一体化 | 需要 MCP 社区接受 |
| **Phase 3（长期）** | 成为 MCP 核心字段 | 最原生的集成 | 需要规范级别的变更 |

**Phase 1 的技术方案**：
```
GET https://api.example.com/.well-known/asm
→ 返回该服务器所有工具的 ASM manifest 列表

Agent 的发现流程：
1. GET .well-known/mcp → 能力发现（已有）
2. GET .well-known/asm → 价值发现（ASM 新增）
3. 合并两者 → 完整的服务画像
```

**这个决策在 Discussion #69 的回复里已经用问题形式抛出了**（"Should value metadata live in ToolAnnotations or as a separate endpoint?"），等社区反馈来指导下一步。

### 13.5 Scorer 的设计哲学 — 简单加权 vs 复杂匹配

**问题**：当前的 `score()` 函数是简单的加权平均。够用吗？

**答案：对 v0.2 够用，但 v1.0 需要两阶段设计。**

**当前方案（v0.2）的问题**：
```python
# 当前：纯加权平均
score = w_cost * norm_cost + w_quality * norm_quality + w_speed * norm_speed
```

问题：如果用户说"我要质量 ≥ 80 分的"，但加权平均可能选出一个质量 70 分但成本极低的服务，因为低成本把总分拉上去了。用户的**硬约束**被**软偏好**覆盖了。

**v1.0 方案：Filter → Rank 两阶段**：

```
Stage 1: Filter（硬约束过滤）
  → quality ≥ 0.8 AND latency ≤ 5s AND cost ≤ $0.10/req
  → 不满足任何一条的服务直接淘汰

Stage 2: Rank（软偏好排序）
  → 在通过过滤的服务中，用加权平均排序
  → score = w_cost * norm_cost + w_quality * norm_quality + w_speed * norm_speed
```

**偏好函数的输入格式**：
```json
{
  "constraints": {
    "quality": { "min": 0.8 },
    "latency_p50": { "max": "5s" },
    "cost_per_request": { "max": 0.10 }
  },
  "preferences": {
    "cost": 0.5,
    "quality": 0.3,
    "speed": 0.2
  }
}
```

**为什么 v0.2 只做加权平均就够了**：
- 现阶段的目标是证明"ASM 数据可以驱动自动决策"
- Filter → Rank 是工程优化，不是协议创新
- 论文/demo 中用简单加权就够了，在 Discussion/Limitations 里提到两阶段设计作为 future work

### 13.6 最小可行生态 — 需要多少 Provider？

**问题**：ASM 需要覆盖多少个服务才有实际价值？

**答案：3 个同类服务就够启动，20-30 个跨品类服务就能做有说服力的 demo。**

**核心逻辑**：ASM 的价值在于**比较**。只有 1 个服务时没有比较的意义，2 个勉强可以，**3 个同类服务是最小闭环**。

**最小可行 demo 的构成**：

| 品类 | 需要几个服务 | 候选 |
|------|-----------|------|
| LLM Chat | 3 | Claude Sonnet 4, GPT-4o, Gemini 2.5 Pro |
| Image Gen | 3 | FLUX 1.1 Pro, DALL-E 3, Imagen 3 |
| Video Gen | 2 | Veo 3.1, Kling |
| TTS | 2 | ElevenLabs, OpenAI TTS |
| GPU Compute | 2 | Replicate, RunPod |
| Embedding | 2 | text-embedding-3-large, Voyage AI |
| **合计** | **~14** | — |

**14 个服务就能覆盖 6 个品类，每个品类至少 2-3 个选择**——这已经足够做一个有说服力的 end-to-end demo。

**扩展路径**：
- Phase 1（demo）：14 个手动填充
- Phase 2（黑客松）：扩展到 30 个，覆盖更多边缘品类
- Phase 3（开源后）：社区贡献，目标 100+

**不需要等"所有 provider 都支持 ASM"才能证明价值。14 个服务 + 3 个真实用户场景 + 1 次 end-to-end 演示 = 足够证明 concept。**

## 十四、Scorer 深化 + 剩余调研方向（2026-04-03）

> 要让人信服"ASM 不是空中楼阁"，Scorer 的学术深度和 Provider 激励的经济学论证是两根支柱。

### 14.1 Scorer 的学术定位 — ASM vs LLM Routing 的关键区别

发现了两个 ASM scorer 必须搞清楚关系的学术方向：

#### A. LLM Routing（算法层，动态，运行时决策）

**代表工作**：
- [RouteLLM](https://github.com/lm-sys/RouteLLM)（LMSYS，4.8K Stars）— 用矩阵分解/BERT/Elo 等方法做 LLM 路由，成本降 85%，保 95% GPT-4 性能
- [Dynamic Model Routing and Cascading Survey](https://arxiv.org/abs/2603.04445)（2026.02，Trinity College Dublin + Huawei）— LLM 路由的最全综述
- [MasRouter](https://aclanthology.org/2025.acl-long.757/)（ACL 2025）— 多 Agent 系统的 LLM 路由

**RouteLLM 的核心方法**：
- 路由器根据用户 prompt 计算"强模型获胜率"
- 如果获胜率 > 阈值 → 路由到强模型（贵），否则 → 弱模型（便宜）
- 4 种路由算法：矩阵分解（推荐）、加权 Elo、BERT 分类器、LLM 分类器

**ASM Scorer vs LLM Routing 的本质区别**：

| 维度 | LLM Routing（RouteLLM 等） | ASM Scorer |
|------|--------------------------|-----------|
| **决策时机** | 运行时（每次请求实时计算） | 选择时（Agent 接到任务时一次性决策） |
| **输入数据** | 用户 prompt 的内容/难度 | 服务的结构化元数据（pricing/quality/SLA） |
| **决策逻辑** | ML 模型预测"哪个模型能更好地回答这个问题" | 偏好函数计算"哪个服务的性价比最符合用户需求" |
| **适用范围** | 仅限 LLM（同品类路由） | 跨品类（LLM/图像/视频/语音/GPU 都能选） |
| **需要训练** | 是（需要偏好数据训练路由器） | 否（纯规则+数学，零训练数据） |
| **互补关系** | RouteLLM 在"选好品类后"做细粒度路由 | ASM 在"跨品类"层面做粗粒度选择 |

**一句话区分**：RouteLLM 回答"这个问题该用 GPT-4 还是 Mixtral"，ASM 回答"这个任务该用 LLM、图像生成还是视频生成，用哪家的"。

**论文中可以这样写**：
> "LLM routing systems like RouteLLM (Ong et al., 2024) operate at the intra-category level, dynamically selecting among LLMs based on query difficulty. ASM operates at the inter-category level, selecting among heterogeneous services based on structured economic metadata. The two are complementary: ASM selects the service category and provider, then RouteLLM (if applicable) selects the specific model within that provider."

#### B. MCDM（多准则决策，ASM Scorer 的理论根基）

ASM Scorer 本质上是一个 **Multi-Criteria Decision Making (MCDM)** 问题。云服务选择领域已有大量研究：

**经典方法**：
| 方法 | 全称 | 核心思想 | 适合 ASM 吗？ |
|------|------|---------|-------------|
| **TOPSIS** | Technique for Order Preference by Similarity to Ideal Solution | 计算每个候选方案到"理想方案"和"最差方案"的距离，选最接近理想方案的 | ✅ 很适合，比加权平均更有说服力 |
| **AHP** | Analytic Hierarchy Process | 用户通过两两比较确定权重 | 权重获取太复杂，不适合 agent 自动化 |
| **PROMETHEE** | Preference Ranking Organization Method | 偏好流分析 | 适合学术，工程复杂度高 |
| **CODAS** | Combinative Distance-based Assessment | 欧式距离 + 曼哈顿距离组合 | 适合论文消融实验 |

**ASM Scorer v1.0 的推荐方案：Filter + TOPSIS**

```
Stage 1: Filter（硬约束）
  → 淘汰不满足最低要求的服务

Stage 2: TOPSIS（排序）
  → 构建决策矩阵（m 个候选服务 × n 个评估维度）
  → 归一化 → 加权 → 计算到正理想解和负理想解的距离
  → 输出排序 + 各维度得分明细 + 选择理由
```

**为什么 TOPSIS 比简单加权平均好**：
1. 加权平均可能被一个极端高分维度"带偏"——比如一个服务免费但质量极差，加权平均可能给高分
2. TOPSIS 同时考虑"离最好的有多近"和"离最差的有多远"，更稳健
3. TOPSIS 在学术界被广泛接受，论文引用量巨大，审稿人看到会加分

**论文的消融实验设计**：
- Baseline: 随机选择
- Method 1: 简单加权平均（v0.2 当前方案）
- Method 2: Filter + 加权平均
- Method 3: Filter + TOPSIS（v1.0 推荐方案）
- 评估指标：用户满意度模拟、成本节约率、质量达标率

### 14.2 博弈论 — Provider 虚报问题的学术解

**重要论文发现**：[Pay for the Second-Best Service: A Game-Theoretic Approach Against Dishonest LLM Providers](https://arxiv.org/abs/2511.00847)（WWW 2026）

**这篇论文直接解决了 ASM 的信任问题。** 核心内容：

**问题**：LLM Provider 可能：
1. "偷梁换柱"——宣称用 GPT-4 实际用 Mixtral（省成本）
2. "注水"——在回答中填充无意义 token（多收费）

**解法**：近似激励相容机制（Approximately Incentive-Compatible Mechanism）
- 用户将 T 个查询迭代委托给多个 provider
- 机制设计使得 provider "说真话"比"造假"更赚钱
- 数学保证：损失控制在 O(T^{1-ε} log T) 范围内

**与 ASM 的关系**：
- 该论文解决的是**运行时**的 provider 造假问题（每次请求级别）
- ASM 解决的是**选择时**的信息不对称问题（服务元数据级别）
- **两者互补**：ASM 提供选择依据 → 该机制保证选择后的执行诚实性

**ASM 论文中的引用方式**：
> "While ASM addresses pre-selection information asymmetry through structured metadata, runtime dishonesty (model substitution, token stuffing) requires complementary mechanisms. Cao et al. (2026) propose an approximately incentive-compatible mechanism for iterative service delegation, achieving O(T^{1-ε} log T) regret. ASM's signed receipt integration (Section X) bridges these two layers: ASM declares expected quality, receipts verify actual delivery, and game-theoretic mechanisms govern the payment."

## 十五、致命问题预案 — 评委/审稿人一定会问的 7 个问题（2026-04-03）

> 如果你明天站在 Anthropic AI Hackathon 评委面前，这些问题答不上来就会被打回。

### 15.1 "为什么不让 LLM 直接读定价页面？"

**这是最致命的质疑。** 如果答不好，整个 ASM 的存在意义就被否定了。

**回答框架（5 个维度）**：

| 维度 | 让 LLM 读网页 | 用 ASM |
|------|-------------|-------|
| **成本** | 每次选服务消耗数千 token 读 3-5 个页面 | 解析结构化 JSON，成本接近零 |
| **准确性** | 定价页面给人看，格式五花八门，LLM 提取复杂定价（阶梯价+条件+多维度）准确率低 | 结构化字段，零解析错误 |
| **时效性** | LLM 训练数据有截止日期，价格变了 LLM 不知道 | manifest 有 `updated_at` 和 `ttl`，保证新鲜度 |
| **可比性** | LLM 的"比较"是概率性的，不可复现，两次问可能得不同答案 | scorer 是确定性数学函数，可审计、可解释、可复现 |
| **规模化** | 选 1 个服务要读 3 个页面，选 100 个呢？成本指数级增长 | Registry 查询恒定成本，不随候选数增长 |

**一句话杀手回答**：
> "Asking an LLM to read pricing pages is like asking a human to comparison-shop by reading restaurant menus in different languages. ASM is the standardized nutrition label — instant, accurate, automatable."

### 15.2 "真实 Agent 工作流里 ASM 在哪一步？"

**完整时序图**：

```
用户 → Agent: "帮我把这段视频加字幕"

Agent 内部流程：
  1. 任务解析 → 识别需要"字幕生成"能力（taxonomy: ai.video.subtitle）
  2. 查询 ASM Registry → GET .well-known/asm?taxonomy=ai.video.subtitle
     ← 返回 3 个服务的 manifest（含 pricing/quality/SLA）
  3. 加载用户偏好 → { constraints: {quality ≥ 0.7}, preferences: {cost: 0.5, quality: 0.3, speed: 0.2} }
  4. ASM Scorer 计算 → Filter（淘汰不达标的）→ TOPSIS（排序）
     ← 输出：Service B 得分最高（0.769），理由：性价比最优
  5. Agent 确认选择 → 向用户展示"我选了 B，因为..."（可选，取决于授权模式）
  6. 调用服务 → 通过 MCP tool call 调用 Service B
  7. 生成 Receipt → Agent Receipts SDK 记录实际花费、延迟、质量
  8. 更新 Trust Score → 对比 ASM 声明 vs Receipt 实际
  9. 返回结果 → 用户拿到字幕文件

整个流程中 ASM 的触点：Step 2（数据查询）、Step 4（决策计算）、Step 8（信任更新）
```

### 15.3 "没有 ASM 的服务怎么处理？"

```
Agent 查 ASM Registry
  ├── 有 Provider 声明的 ASM → 优先使用，trust 最高
  ├── 有 Registry 自动抓取的 ASM → 可用，标记 self_reported: false，trust 中等
  ├── 没有 ASM 但有公开 API → 降级为"未评级"，排序中降权但不排除
  └── 完全未知的服务 → 排最后，除非用户明确指定"我就要用这个"
```

**关键设计**：ASM 的 `data_source` 字段（v0.3 新增建议）：
- `"provider_declared"` — Provider 自己填的
- `"registry_crawled"` — Registry 从公开页面抓取的
- `"community_contributed"` — 社区贡献的
- `"unknown"` — 无数据

### 15.4 "治理模型是什么？谁来维护？"

参考 MCP 自身的 CWG（Community Working Groups）治理结构：

| 治理事项 | 机制 | 类比 |
|---------|------|------|
| **Taxonomy 新增/修改** | GitHub PR + 社区 review + 至少 2 个真实服务使用该分类 | IANA Media Types 注册 |
| **Billing dimension 新增** | 同上，需提供真实定价案例证明必要性 | HTTP Status Code 注册 |
| **Schema 版本升级** | SEP 流程（MCP 的规范增强提案） | W3C Recommendation 流程 |
| **质量数据裁定** | ASM 不做裁定，只提供透明度。Signed Receipts 提供事后验证，市场自然淘汰造假者 | 类似 Yelp——平台不裁定菜好不好吃，用户评价说话 |
| **Registry 运维** | 初期由 ASM 开源社区维护，长期可由 AAIF（Linux Foundation 下的 Agent 开放标准组织）托管 | 类似 npm registry 由 npm Inc 维护 |

### 15.5 "隐私问题——查询会暴露用户意图？"

**问题**：Agent 查 ASM Registry 时，查询内容（taxonomy + 偏好权重）暴露了用户正在做什么以及愿意花多少钱。

**回答**：

| 方案 | 描述 | 适用阶段 |
|------|------|---------|
| **本地 Registry** | Agent 本地缓存 manifest，不联网查询 | v1.0 可实现 |
| **模糊查询** | 只查 taxonomy 前缀（`ai.video.*`），不暴露具体子类和偏好 | v0.2 即可 |
| **差分隐私** | 在查询中注入噪声 | 长期研究方向 |
| **端侧 Scorer** | Registry 返回所有符合品类的 manifest，scorer 在本地运行，偏好不离开设备 | 推荐方案 |

**推荐方案**：Registry 只做"给我这个品类的所有服务"的粗查询，scorer 在 Agent 本地运行。偏好函数和权重永远不发送给 Registry。

### 15.6 "10 万个服务时性能怎么办？"

**短期（<1000 服务）**：全量返回 + 本地过滤，无性能问题。

**中期（1K-100K）**：
- Registry 支持按 taxonomy 前缀索引查询（`?taxonomy=ai.video.*`）
- 返回结果分页（cursor-based pagination）
- Agent 端缓存（respect `ttl` 字段）

**长期（>100K）**：
- Registry 支持 pre-filter 查询参数：`?taxonomy=ai.video.subtitle&max_cost=0.10&min_quality=0.7`
- 返回已过滤的候选集（通常 <20 个），Agent 端做最终排序

**这不是 ASM 协议层面的问题，是 Registry 实现层面的问题。** 协议只定义 manifest 格式，Registry 的查询优化是工程问题。

### 15.7 "多币种/国际化怎么处理？"

**当前设计**：`currency` 字段默认 `"USD"`，支持 ISO 4217 货币代码。

**扩展方案**：
```json
{
  "pricing": {
    "billing_dimensions": [
      {
        "dimension": "input_token",
        "unit": "per_1M",
        "cost_per_unit": 3.00,
        "currency": "USD",
        "regional_pricing": {
          "CN": { "cost_per_unit": 20.00, "currency": "CNY" },
          "EU": { "cost_per_unit": 2.80, "currency": "EUR" }
        }
      }
    ]
  }
}
```

**v0.2 不需要做这个**，在 Discussion/Future Work 中提一句就够了。
