## 1. Schema v0.3 — 新增 4 个字段

**文件**: `schema/asm-v0.3.schema.json`（新建）

在 v0.2 Schema 基础上，新增了 4 个顶层字段，用于支持 Signed Receipts 集成和清单数据新鲜度管理：

### 新增字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `updated_at` | `string` (date-time) | ISO 8601 时间戳，标记清单最后更新时间。Agent 据此判断数据是否过期。 |
| `ttl` | `integer` | 缓存有效期（秒）。0 = 每次都重新获取。推荐：稳定服务 3600（1小时），价格波动大的服务 300（5分钟）。 |
| `receipt_endpoint` | `string` (uri) | Signed Receipts 端点 URL。服务执行后，Agent 向此端点 POST 以获取加密签名的交付收据。 |
| `verification` | `object` | 验证配置对象，包含以下子字段： |
| ↳ `verification.protocol` | `string` | 验证协议：`signed-receipts-acta`（IETF ACTA，推荐）、`w3c-vc`（W3C 可验证凭证）、`custom` |
| ↳ `verification.public_key` | `string` | PEM 或 JWK 格式的公钥，用于验证收据签名 |
| ↳ `verification.public_key_url` | `string` (uri) | 公钥获取 URL（支持密钥轮换） |
| ↳ `verification.receipt_schema_version` | `string` | 收据 schema 版本号 |

### 设计决策

- **v0.2 Schema 保持不变**：v0.3 是独立文件，不破坏现有清单的兼容性
- `asm_version` 从 `"0.2"` 升级为 `"0.3"`
- `receipt_endpoint` 和 `verification` 放在顶层（而非嵌套在 `payment` 中），因为它们属于信任层而非支付层
- `updated_at` + `ttl` 组合解决了"清单数据多久刷新一次"的问题

## 2. Trust Delta Scoring with Exponential Decay

**文件**: `scorer/scorer.py`（修改，新增约 200 行）

在现有评分引擎中新增了 **Stage 3: Trust Delta Scoring** 模块，实现了基于执行收据的信任评分机制。

### 核心公式

**信任偏差（Trust Delta）**：
```
trust_delta(declared, actual) = |declared - actual| / declared
```
- 0.0 = 完全匹配（声明值 = 实际值）
- 0.5 = 50% 偏差
- 1.25 = 声明延迟 200ms，实际 450ms

**指数衰减权重**：
```
w(t) = exp(-ln(2) × age / half_life)
```
- 最近的收据权重最高（w ≈ 1.0）
- 1 周前的收据权重减半（w = 0.5）
- 1 个月前的收据几乎忽略（w ≈ 0.05）

**综合信任分数**：
```
trust_score = 1 - mean(weighted_deltas_per_dimension)
```

### 新增数据结构

- `ReceiptRecord` — 单次执行收据，记录实际交付指标（延迟、质量、可用性、成本）
- `TrustScore` — 计算后的信任分数，包含各维度偏差分解、置信度、推理说明

### 新增函数

| 函数 | 说明 |
|------|------|
| `compute_trust_delta(declared, actual)` | 计算单个维度的信任偏差 |
| `exponential_decay_weight(timestamp, now, half_life)` | 计算基于时间的指数衰减权重 |
| `compute_trust_score(service, receipts, ...)` | 综合计算服务的信任分数 |
| `adjust_scores_with_trust(scored_services, trust_scores, trust_weight)` | 将信任分数融入 TOPSIS/加权平均排名 |

### 信任调整公式

```
final_score = (1 - trust_weight) × original_score + trust_weight × trust_score × confidence
```

- `trust_weight` 默认 0.2（信任占最终分数的 20%）
- `confidence` 基于收据数量渐近增长：`1 - exp(-n/5)`，10 条收据时约 0.86，20 条时约 0.98
- 无收据数据的服务不受影响（trust=0.5, confidence=0）

## 3. Signed Receipts Integration Demo

**文件**: `demo/receipts_demo.py`（新建）

一个完整的端到端演示，展示 ASM ↔ Signed Receipts 信任链的工作流程。

### 演示内容（4 个部分）

**Part 1: Trust Delta 基础**
- 展示信任偏差公式的计算示例
- 不同偏差程度的颜色标注（绿色=好，黄色=警告，红色=严重）

**Part 2: 指数衰减权重**
- 可视化不同时间点的收据权重
- 展示半衰期效果（1天前 0.91，1周前 0.50，1月前 0.05）

**Part 3: 完整信任管线**
- 使用 3 个 LLM 服务（Claude Sonnet 4、Gemini 2.5 Pro、GPT-4o）
- 模拟不同诚信度：诚实（honesty=1.0）、夸大（honesty=1.8）、轻微膨胀（honesty=1.2）
- 每个服务生成 20 条模拟收据
- 展示 IETF ACTA 格式的签名收据样本
- 计算各服务的信任分数和各维度偏差
- 用信任分数重新排名，展示排名变化

**Part 4: v0.3 清单示例**
- 展示包含新字段的完整 v0.3 清单 JSON

### 运行方式

```bash
python3 demo/receipts_demo.py
```

无需任何外部依赖，纯 Python 实现。使用固定随机种子（seed=42）保证结果可复现。

## 5. arXiv 论文 Section 4-8（完成）

**文件**: `paper/asm-paper-draft.md`（修改，新增约 450 行）

论文从 "Draft — Sections 1-3" 升级为 "Draft — Complete (Sections 1-8)"。新增 5 个章节：

### Section 4: Protocol Design

详细阐述了 ASM 协议的 5 个核心设计：

- **4.1 Manifest Schema**：最小核心 + 最大可选的设计哲学（3 个必填字段 + 5 个可选模块），v0.3 新增的 4 个字段（`updated_at`、`ttl`、`receipt_endpoint`、`verification`）
- **4.2 Hierarchical Taxonomy**：18 类分层分类体系，支持前缀查询（如 `ai.llm.*`），正则验证格式
- **4.3 Pricing Engine**：多维定价模型（12 种计费维度、3 种单位粒度），支持阶梯定价和条件定价，成本归一化公式 $c_{\text{repr}} = 0.3 \cdot c_{\text{input}} + 0.7 \cdot c_{\text{output}}$
- **4.4 Quality and Trust Model**：三层信任架构的完整数学形式化——信任偏差公式、指数衰减权重、置信度函数
- **4.5 Integration Architecture**：三阶段 MCP 集成路径（独立端点 → `x-asm` ToolAnnotations → 原生 MCP 字段），Signed Receipts 的 W3C Verifiable Credentials 格式

### Section 5: Reference Implementation

描述了三个实现组件：

- **5.1 Scoring Engine**：Python 评分引擎的三阶段管线（约束过滤 → TOPSIS → 信任调整），质量归一化表（Elo、FID、MOS 等不同量纲的统一处理）
- **5.2 MCP Server**：TypeScript MCP 服务器的 6 个工具（`asm_list`、`asm_get`、`asm_query`、`asm_compare`、`asm_score`、`asm_taxonomies`），语言无关的架构决策
- **5.3 Demonstration Scripts**：两个 demo 脚本的场景覆盖

### Section 6: Evaluation

四个维度的评估：

- **6.1 定价异构性覆盖**：14 个服务的定价模型汇总表，8 种计费模型、3 种单位粒度，验证 Schema 的表达能力
- **6.2 评分准确性**：4 种偏好配置下 3 个 LLM 服务的 TOPSIS 排名对比表，证明不同偏好产生不同最优选择
- **6.3 信任偏差有效性**：诚实（1.0）、夸大（1.8）、轻微膨胀（1.2）三种诚信度的信任分数对比，展示不诚实服务被正确降级
- **6.4 协议开销**：Schema 14.5KB、清单 ~1.2KB、评分 <5ms、每清单 ~300 tokens——开销可忽略

### Section 7: Discussion

- **局限性**：静态声明、质量归一化信息损失、信任冷启动、对抗鲁棒性、分类完整性
- **信任机制对比**：与 Cao et al.（运行时诚实性）、Agent Receipts（加密基础设施）、MCP ToolAnnotations（"hints should not be trusted"）的互补关系
- **市场设计视角**：ASM 作为 AI 服务市场的信息基础设施（类比营养标签）
- **未来方向**：实时定价、联邦注册中心、学习偏好、多服务组合优化、激励对齐

### Section 8: Conclusion

总结 4 项核心贡献：最小可扩展 Schema、三层信任模型、两阶段选择引擎、完整参考实现。

## 6. SEP Proposal to MCP Specification（完成）

**文件**: `sep/sep-asm-service-value.md`（新建，约 290 行）

一份完整的 MCP 规范增强提案（Specification Enhancement Proposal），提议在 MCP 的 `ToolAnnotations` 中添加 `x-asm` 服务价值注解。

### 提案结构

| 章节 | 内容 |
|------|------|
| **Abstract** | 一句话概述：为 MCP ToolAnnotations 添加结构化的定价、质量、SLA 和信任信息 |
| **1. Motivation** | MCP 描述"工具能做什么"但不描述"工具值多少"；无结构化数据时盲选可导致 3-10× 成本超支；AWS Marketplace MCP Server 验证了需求但是封闭方案 |
| **2. Proposal** | `x-asm` 字段的完整 TypeScript 接口定义 + FLUX 1.1 Pro 的完整示例 JSON |
| **3. Design Decisions** | 为什么选 ToolAnnotations 扩展（vs 独立资源 vs 顶层字段）、为什么结构化定价（vs 自由文本）、为什么 `self_reported` 标志、为什么指数衰减 |
| **4. Backward Compatibility** | 完全向后兼容，三阶段迁移路径：`x-asm` → `asm` → native MCP fields |
| **5. Implementation** | 指向现有参考实现（Schema、Scorer、MCP Server、14 个清单、2 个 Demo） |
| **6. Security Considerations** | 信任（hints not guarantees）、价格操纵（Signed Receipts 检测）、信息泄露（公开元数据） |

### 核心提案内容

在 MCP `ToolAnnotations` 中添加可选的 `x-asm` 字段：

```typescript
interface ToolAnnotations {
  // 现有字段不变
  readOnlyHint?: boolean;
  destructiveHint?: boolean;
  // 新增
  "x-asm"?: ServiceValueAnnotation;
}
```

`ServiceValueAnnotation` 包含：`service_id`（必填）、`taxonomy`（必填）、`pricing`（可选）、`quality`（可选）、`sla`（可选）、`verification`（可选）。
